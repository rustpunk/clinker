# Error E325: Combine output exceeded max_output_rows

## What it means

A Combine node carried an opt-in `max_output_rows` cap, and the number of
rows the combine would emit reached that cap. Clinker stops the run with
E325 rather than emitting a truncated or partial result.

`max_output_rows` is a **result-size runaway guard**: an unbounded or
mis-specified join predicate can turn a small pair of inputs into an
explosively large output (a near cross product). The cap lets you assert
"this combine should never emit more than N rows; if it would, something is
wrong — stop." It is opt-in and defaults to unlimited.

This is deliberately **not** an out-of-memory error. The combine's output
sort spills to disk to stay inside `memory.limit`, so a runaway join can
produce a correct-but-huge result while comfortably inside its RSS
envelope. E325 fires on the row **count**, not on bytes, and is independent
of the memory budget.

The runtime diagnostic names the combine node and the configured cap:

```
E325 combine 'enrich': output exceeded max_output_rows of 1000 rows. This
is a result-size runaway guard, not an out-of-memory condition: the combine
would emit more than the configured cap, so the run stops rather than
truncating to a partial result. Raise max_output_rows if the large result
is expected, or tighten the combine predicate.
```

## Example

```yaml
# A combine whose predicate matches far more pairs than expected — e.g. a
# range join over a hot key — would emit millions of rows. The cap stops it
# at 1000 instead of letting a mistaken predicate flood the sink.
- type: combine
  name: enrich
  input:
    orders: orders
    prices: prices
  config:
    where: "orders.ts >= prices.effective_from"
    cxl: |
      emit order_id = orders.order_id
      emit price = prices.amount
    propagate_ck: driver
    max_output_rows: 1000
```

```yaml
# Corrected: the large result was expected (a legitimate many-to-many
# expansion), so raise the cap to fit it — or omit max_output_rows entirely
# to leave the combine uncapped.
    max_output_rows: 5000000
```

## How to fix

Pick the remediation that matches your intent:

1. **The large result is wrong.** Tighten the combine `where:` predicate so
   it matches the intended pairs — add an equality conjunct, narrow the
   range bounds, or fix a key that is hotter than expected.

2. **The large result is expected.** Raise `max_output_rows` to a ceiling
   that fits the real output, or remove the field to leave the combine
   uncapped (the default).

3. **You only need a bounded sample per driver.** Switch the combine to
   `match: first` or `match: collect`, which emit at most one row per driver
   row, instead of `match: all`.

## Technical context

E325 is a runtime diagnostic. Each combine strategy keeps a running count of
the rows it has emitted and, at its output-emit chokepoint, compares that
count against the configured cap before committing more. When the output
would exceed the cap the combine returns
`PipelineError::CombineOutputCapExceeded` and the CLI exits with the
config/semantic status code.

The cap is **strategy-agnostic**: whichever join strategy the planner picks —
hash build-probe, grace-hash, sort-merge, or the IEJoin block-band (including
its bounded nested-loop fallback for a dense pair) — every emitted row passes
one such check, so the cap covers every match mode (`all`, `first`,
`collect`) and the deferred `on_miss` rows. Recoverable rows routed to the
dead-letter queue never reach the emit chokepoint and are not counted. Under
`match: collect` the counted rows are the per-driver output rows (each
carrying its collected array), not the individual collected elements.

The cap is kept distinct from the memory budget (E310) on purpose: a
permissive join can be perfectly bounded in memory (the output sort spills)
yet still produce a result far larger than intended. Row count and byte
footprint are different failure axes, so they carry different diagnostics.
