# Error E326: Combine decimal range key out of fixed-point range

## What it means

A combine node's range/inequality predicate compares `decimal` values,
and one range key fell outside the exact fixed-point range the numeric
inequality-join axis can represent. Clinker stops the run rather than
dropping the row or emitting a wrong match.

The inequality-join axis places every decimal on a shared fixed-point
grid — the value scaled to `10^18` — and carries it as a 128-bit
integer so both sides of the comparison sort in one exact, order-
preserving space. Two conditions fall off that grid:

- **Overflow** — the value's integer magnitude exceeds roughly
  `1.7 × 10^20`, so scaling it onto the grid overflows the 128-bit key.
- **Truncation** — the value carries more than 18 fractional digits, so
  placing it on the grid would silently discard precision and could flip
  a comparison.

The runtime diagnostic names the combine node and the offending value:

```
E326 combine 'assign_tier': range key 79228162514264337593543950335 is outside the exact fixed-point range the decimal inequality-join axis supports — its magnitude exceeds ~1.7e20 or it carries more than 18 fractional digits. Rescale or narrow the compared values. See: clinker explain --code E326
```

## Example

```yaml
# A monetary band join. The tier floors/ceilings and the amount are all
# decimals — supported — but one amount here is astronomically large.
nodes:
  - type: combine
    name: assign_tier
    input:
      payments: payments
      tiers: tiers
    where: "payments.amount >= tiers.floor and payments.amount < tiers.ceiling"
    match: first
    cxl: |
      emit payments.id, tiers.name
```

If `payments.amount` holds a value beyond the representable range — a
magnitude past ~1.7e20, or more than 18 fractional digits — the run
stops with E326 instead of silently emitting nothing for that row.

## How to fix

1. **Rescale the compared columns.** Realistic monetary magnitudes
   (billions, even quintillions) with a handful of decimal places sit
   comfortably inside the grid. A value that overflows it is almost
   always a units error (e.g. a raw micro-denomination that should be
   divided down) or bad data — correct it upstream.

2. **Reduce the fractional precision** to at most 18 digits on the range
   key. Declare the source column with a `scale:` at or below 18, or
   round the value before the combine, when the extra digits are not
   meaningful to the comparison.

3. **Narrow the range key** to the magnitude the join actually needs. If
   only a bounded sub-range of a very wide column participates in the
   band, filter or transform it to that range before the combine.

## Technical context

E326 is a runtime diagnostic raised during the combine's key-extraction
scan, before any range matching runs. It is distinct from the plan-time
combine predicate errors (E303/E305/E313) and from the E310 memory-budget
abort: the predicate is valid and the run is within its memory envelope,
but a specific data value cannot be represented exactly on the join axis.

The axis deliberately fails loud here rather than dropping the row or
projecting the value lossily — a monetary comparison that silently loses
precision or drops a row is a correctness hazard. The executor returns
`PipelineError::CombineRangeKeyOutOfRange` and the CLI exits as a
configuration/runtime failure.

Integer and finite-float range keys, and mixed integer/float range
comparisons, are always representable and never raise E326.
