# Warning W302: Pure-equi combine with all small inputs — consider InMemoryHash

## What it means

The planner detected a combine node where the where-clause is
pure equi-join (every cross-input conjunct is `==`) and every
input's estimated cardinality fits entirely in memory. The planner
will run the combine with the default HashBuildProbe strategy, but
pinning it to `InMemoryHash` (when it becomes configurable) avoids
the spill infrastructure's bookkeeping overhead.

## Example

```yaml
# Warning: all inputs small + pure-equi predicate
nodes:
  - type: source
    name: products
    estimated_rows: 500                   # small
    config: { type: csv, path: products.csv, schema: [...] }
  - type: source
    name: categories
    estimated_rows: 50                    # tiny
    config: { type: csv, path: categories.csv, schema: [...] }

  - type: combine
    name: enrich
    input:
      products: products
      categories: categories
    where: "products.category_id == categories.id"   # pure-equi
    cxl: |
      emit products.id, categories.name
```

## How to fix

This is an informational warning — the pipeline runs correctly, it
is just a performance hint. Options:

1. **Ignore it.** HashBuildProbe handles small inputs fine; the
   spill infrastructure overhead is small.

2. **Pin the strategy** (once a `strategy:` config field lands —
   currently planner-only):
   ```yaml
   - type: combine
     name: enrich
     strategy: in_memory_hash              # future: pin the plan
     ...
   ```

3. **Let the planner upgrade automatically.** As cost-based
   selection improves, W302 cases may start picking InMemoryHash
   without user intervention.

## Technical context

InMemoryHash is a specialized equi-join strategy that omits spill
plumbing, overflow partitions, and the memory-budget integration
required by GraceHash. It is sound only when every input fits
entirely in memory and the predicate is pure equality. W302 flags
the pattern where the user could safely opt in but the planner
defaults to HashBuildProbe for broad applicability. Per
RESOLUTION W-18, this is distinct from W305/W306 (which flag
planner-hint gaps) — W302 is specifically about pinning in-memory
vs spillable strategies when cardinality makes spill unnecessary.
