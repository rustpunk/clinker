# Error E358: malformed multi-value declaration on a source

## What it means

A source's `split_to_rows` or `split_values` block does not describe a workable
shape. These cases raise this diagnostic:

1. **A `split_to_rows` field is declared twice.** One entry already fans every
   occurrence of that element out to its own record; a second entry would fan
   the same group out again, multiplying the record count for no reason.

2. **Two `split_to_rows` fields nest on an XML source** â€” one names an element
   that sits inside the other (`Item` and `Item.part`). The XML reader assigns
   each extracted element to one occurrence group by document position, so a
   nested pair leaves the inner group's membership ambiguous.

3. **Two `split_to_rows` fields nest on a JSON source and the outer one uses
   `mode: extract`.** Entries apply in declaration order over the flattened
   record, so a nested pair composes â€” but only when the outer entry keeps the
   group's dotted path. `extract` lifts the occurrence's own keys to the top
   level, which removes the path the inner entry addresses; the inner entry then
   matches nothing and `keep_empty`'s default passes the record through
   unfanned. Declare the outer entry `mode: split` for the two-level expansion.

4. **A `split_to_rows` or `split_values` block is declared on a format whose
   reader is never handed it** â€” `csv`, `fixed_width`, or any segment format.
   Only the JSON and XML readers receive these two blocks, over a file path or a
   `rest` response body alike, so anywhere else the declaration is a silent
   no-op: the run emits one record per input row where the author asked for one
   per occurrence.

5. **A `split_values` field is declared twice**, with no way to say which
   delimiter wins.

6. **A `split_values` field is not declared `multiple: true`**, or names no
   schema column at all. Splitting text on a delimiter produces several values,
   and only a multi-value column can hold them; without the schema declaration
   the column is typed as single-valued while carrying an array.

7. **A `split_values` entry declares an empty delimiter.** Splitting on the
   empty string matches at every character boundary, turning one cell into a
   run of single characters bracketed by two empty values.

8. **Two `split_to_rows` entries write the same `position_column`.** Each entry
   numbers its own occurrences, so one index would silently overwrite the other.

9. **One field appears in both `split_to_rows` and `split_values`.** Fanning a
   field out to rows consumes it; parsing it in-cell keeps it and makes it
   multi-valued. The two leave no single shape for the column.

10. **A fanned-out field is also declared `multiple: true`.** The schema
    attribute collects the field's occurrences into one array on one record; the
    fan-out spends them one per record. Declaring both asks for two shapes at
    once.

11. **A `split_values` entry names a column's exposed name rather than the input
    field it reads.** Both split declarations run against the names the DOCUMENT
    carries, before the schema's renaming applies, so a column declared
    `source_name: CostCentres` is addressable only as `CostCentres`. Naming
    `cost_centres` would compile and then never match.

12. **A `split_to_rows` field path has an empty segment** (`Items..Item`,
    `.Item`). No flattened field name can have one, so the entry would fan
    nothing out.

```
source 'orders': `split_to_rows` fields 'Item' and 'Item.part' nest â€” declared
fan-out fields must name disjoint (non-nested) element groups
```

## Example

<!-- explain-test: expect-code=E358 -->
```yaml
# Rejected: the two declared fan-out fields nest.
nodes:
  - type: source
    name: orders
    config:
      name: orders
      type: xml
      path: ./orders.xml
      options:
        record_path: Orders/Order
      split_to_rows:
        - Item
        - Item.part
      schema:
        - { name: "Item.part.code", type: string }
```

```yaml
# Rejected: `tags` is split into several values but declared single-valued.
      split_values:
        - field: tags
          delimiter: ";"
      schema:
        - { name: tags, type: string }
```

```yaml
# Corrected: fan out on one field, and declare the split column multi-value.
# `mode: extract` (the default) lifts the `Item.` prefix off, so the column the
# schema declares is `code`, not `Item.code`.
      split_to_rows:
        - Item
      split_values:
        - field: tags
          delimiter: ";"
      schema:
        - { name: code, type: string }
        - { name: tags, type: string, multiple: true }
```

## How to fix

1. **Declare each field once** in `split_to_rows` and once in `split_values`.

2. **On an XML source, pick one level to fan out on.** If you need both, fan
   out on the outer element and read the inner one's fields off the resulting
   records, or split the work across two source nodes.

3. **On a JSON source, declare the outer entry `mode: split`** when two entries
   nest. That keeps each occurrence's fields under their dotted path, which is
   what the inner entry addresses.

4. **Delete a block the format's reader never receives.** `split_to_rows`
   fan-out is read by the JSON and XML readers; `split_values` in-cell parsing is
   read by the JSON and XML readers and by the single-schema CSV and fixed-width
   readers. A block on any other format â€” or on a multi-record CSV / fixed-width
   source, whose backend does not consume it â€” is a no-op this gate rejects.

5. **Add `multiple: true`** to the schema column a `split_values` entry names,
   or drop the entry if the field really does hold one value. If the diagnostic
   says the schema does not declare the field at all, check the spelling against
   the `schema:` block first.

6. **Give a non-empty delimiter.** The default is `;`; omit the key to take it.

7. **Give each fan-out entry its own `position_column`**, or drop it from all
   but one. The indexes are per-entry and cannot share a column.

8. **Choose one treatment per field.** A field either fans out to rows, is
   parsed in-cell into several values, or is collected into an array by
   `multiple: true` â€” declaring two of the three leaves no single shape for the
   column.

9. **Name the input field, not the exposed column.** For a column that declares
   `source_name:`, a `split_values` entry names the `source_name` value.

10. **Write the field path with no empty segment** â€” the record-relative dotted
   form the source's flattened field names use (`line_items`, `Items.Item`).

## Technical context

The disjointness rule comes from how the XML reader tracks occurrences: while
walking a record element it records, per declared field, the half-open range of
extracted fields each occurrence covers, and it keeps at most one occurrence
open at a time. That is sound exactly when declared fields name disjoint element
groups â€” which is why the rule is scoped to that format rather than applied to
every source.

Both split declarations are matched by the readers against physical document
field names, the same rule that hands the readers the physical names of the
`multiple: true` columns. The plan-time check therefore requires the physical
name too: accepting the exposed name here would have compiled a declaration that
could never fire.

This check ran at XML reader construction, as an untyped string error with no
diagnostic code and no YAML span â€” invisible to `clinker explain` and reported
only once a run had already started opening input. It never ran at all on the
JSON path, where an overlapping declaration silently produced a fan-out nobody
asked for. Both formats are validated here instead, at compile, anchored to the
source node.
