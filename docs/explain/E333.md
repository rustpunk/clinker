# Error E333: Staging directory on the same device as a staged source

## What it means

The `storage.staging.dir` you configured sits on the **same physical
device** as a source the staging patterns match. Staging copies a source
file to local disk so the run reads from a stable, fast copy instead of the
original — but if the copy lands on the *same* device as the original, the
copy moves no I/O off that volume and buys nothing while still spending the
time and space to duplicate every matched file. The run is refused at
startup rather than allowed to perform a pointless copy.

The startup diagnostic names both the staging directory and the offending
source:

```
E333 storage.staging.dir /data/staging is on the same physical device as
staged source /data/inputs/orders.csv; staging onto the same device copies
bytes without moving I/O off the source volume — point storage.staging.dir
at a local disk on a different device. See: clinker explain --code E333
```

## Example

```toml
# clinker.toml — staging dir and the matched source share one device.
[storage.staging]
enabled  = true
dir      = "/data/staging"            # same disk as /data/inputs
patterns = ["/data/inputs/*.csv"]
```

```toml
# Corrected: stage onto a different physical device than the inputs.
[storage.staging]
enabled  = true
dir      = "/mnt/fast-ssd/staging"    # a separate disk
patterns = ["/data/inputs/*.csv"]
```

## How to fix

1. **Point `storage.staging.dir` at a different device.** Pick a path on a
   separate physical disk — ideally a faster local one — than the matched
   sources live on. On Linux/macOS the check compares the device id
   (`st_dev`); on Windows it compares the volume serial number.

2. **Narrow the patterns** if a pattern unintentionally matched a source
   that already lives on the staging volume. The same-device rule applies
   only to **matched** sources; an unmatched source reads in place and its
   volume is irrelevant.

## Technical context

E333 is a startup-validation diagnostic raised by `validate_storage_config`
after the plan compiles and before any source-ingest thread spawns. The
same-device decision is made once, by the staging-policy validation, through
the cross-platform `fs_type` facade's `same_device` probe — there is no
second, independent same-device check anywhere in the run. The validator
re-presents that decision with the E333 code so the diagnostic carries a
stable lookup key.

Validating up front fails a misconfigured staging volume while the run is
still cheap to abandon, rather than after every matched file has been
needlessly copied.
