# Error E304: Field not in combine merged row

## What it means

A `where:` or `cxl:` expression references a qualified field
`<qualifier>.<field>` that is not present in the combine's merged row.
Either the qualifier is wrong, the field name is misspelled, or the
upstream schema changed and no longer carries the expected column.

## Example

```yaml
# Invalid: "orders.customer" not in orders schema
nodes:
  - type: source
    name: orders
    config:
      type: csv
      path: orders.csv
      schema:
        - { name: id,         type: string }
        - { name: product_id, type: string }
        - { name: qty,        type: int }

  - type: combine
    name: bad
    input:
      orders: orders
      products: products
    where: "orders.customer == products.id"   # ERROR: orders has no "customer" field
    cxl: |
      emit orders.id
```

```yaml
# Corrected: use a field that actually exists
nodes:
  - type: combine
    name: good
    input:
      orders: orders
      products: products
    where: "orders.product_id == products.id"
    cxl: |
      emit orders.id, products.name
```

## How to fix

Check each `<qualifier>.<field>` reference in your `where:` and `cxl:`
clauses against the schemas of the corresponding inputs. Common causes:
- Misspelled field name (`customer_id` vs `customerid`).
- Wrong qualifier (field exists on `products`, not `orders`).
- Upstream transform dropped the column.

Run `clinker guess` or inspect the upstream node's schema in Kiln to
see what columns are actually in the merged row.

## Technical context

The merged row is the disjoint union of each input's schema, keyed by
qualifier. E304 fires during CXL typecheck of the `where:` and `cxl:`
bodies, after schema propagation for all inputs has completed but
before the planner decomposes the predicate. The primary span points
to the offending reference; secondary spans list the closest-matching
fields in the merged row as "did you mean" candidates.
