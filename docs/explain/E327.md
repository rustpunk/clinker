# Error E327: Combine range conjunct cannot use the range axis

## What it means

A combine node's range/inequality predicate compares two operands whose
types the numeric range axis cannot reduce to one comparable value
without risking a **silent** wrong answer. Clinker rejects the pipeline
at plan time rather than run a join that could drop rows or match the
wrong ones.

The range axis places each key on one shared, order-preserving numeric
scale so the two inputs sort into the same space. Two operand shapes
cannot be placed there safely:

- **An ambiguous `numeric` operand** — the result of `abs`, `min`, `max`,
  or `clamp` over a numeric column. Its runtime value may be an integer
  (which the engine compares exactly) or a float (which is compared
  approximately), and no single axis reduction agrees with the exact
  comparison for every value.
- **A non-orderable pairing** — a string or boolean comparison, or a
  `decimal` compared against a `float`. These do not share an orderable
  numeric axis at all.

The diagnostic names the combine and renders the offending conjunct:

```
combine "assign_tier" range conjunct `abs(a.k) >= abs(b.lo)` cannot use the range axis: an operand has the ambiguous `numeric` type (e.g. from abs/min/max/clamp): its runtime value may be an integer compared exactly or a float, so no single range-axis reduction agrees with the comparison for every value
```

## Example

```yaml
# Invalid: abs() produces the ambiguous `numeric` type, so this range
# comparison cannot be a join axis.
nodes:
  - type: combine
    name: assign_tier
    input:
      a: readings
      b: tiers
    where: "abs(a.k) >= b.floor"
    match: first
    cxl: |
      emit a.id, b.name
```

```yaml
# Corrected: compare a concrete numeric range key. Round/scale the value
# into a real int, float, or decimal column upstream, then join on it.
nodes:
  - type: combine
    name: assign_tier
    input:
      a: readings
      b: tiers
    where: "a.magnitude >= b.floor"      # a.magnitude: a concrete int/float/decimal
    match: first
    cxl: |
      emit a.id, b.name
```

## How to fix

1. **Compare a concrete numeric key.** Materialize the value into a real
   `int`, `float`, or `decimal` column upstream (a Transform that emits
   the computed magnitude), then range-join on that column so both sides
   share one type.

2. **Match the two operand types.** A `decimal` and a `float` cannot be
   mixed on the range axis — cast one side so both are `decimal`, both are
   `float`, or both are `int`. A mixed `int`/`float` range key IS
   supported and does not raise E327.

3. **Move the comparison out of the join predicate.** If the ambiguous or
   non-orderable comparison is a secondary filter rather than the join
   axis, keep an equality or a supported range conjunct as the join key
   and apply the extra condition in the `cxl:` body instead.

## Technical context

E327 is a plan-time diagnostic raised during combine strategy selection,
after the `where:` predicate is decomposed into equality, range, and
residual conjuncts. It is distinct from E305 (no cross-input comparison
at all) and E313 (neither an equality nor a range conjunct): here a range
conjunct exists, but its operand types cannot back a range axis.

The rejection is deliberate fail-loud behavior. The alternative — routing
such a conjunct to the axis anyway — reduces each key through a
per-value encoding that cannot represent the ambiguous or non-orderable
operand, so it would drop every affected row or compare in disjoint
encoding spaces, silently. Integer, float, mixed integer/float, decimal,
date, and datetime range keys all reduce cleanly and never raise E327.
