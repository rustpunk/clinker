# Error E301: Combine input qualifier collides with reserved namespace

## What it means

An input qualifier in a `combine` node's `input:` map uses a name that is
reserved for a CXL system namespace (`$pipeline`, `$window`, `$meta`,
`$doc`) or contains a dot. Qualifiers become two-part path roots inside
CXL expressions (`<qualifier>.<field>`); reserved names and dotted
qualifiers would produce ambiguous three-part references.

## Example

```yaml
# Invalid: qualifier shadows the $meta system namespace
nodes:
  - type: combine
    name: bad
    input:
      meta: orders          # ERROR: "meta" collides with $meta namespace
      products: products
    where: "meta.id == products.id"
    cxl: |
      emit meta.id
```

```yaml
# Invalid: dotted qualifier produces ambiguous 3-part refs
nodes:
  - type: combine
    name: bad
    input:
      "a.b": orders         # ERROR: dotted qualifier
      products: products
    where: "a.b.id == products.id"     # is this (a.b).id or a.(b.id)?
    cxl: |
      emit a.b.id
```

```yaml
# Corrected: rename to a non-reserved, dot-free qualifier
nodes:
  - type: combine
    name: good
    input:
      orders: orders
      products: products
    where: "orders.id == products.id"
    cxl: |
      emit orders.id, products.name
```

## How to fix

Rename the offending qualifier. Pick a short identifier that is not
`pipeline`, `window`, `meta`, or `doc` (with or without a leading `$`)
and does not contain a dot.

## Technical context

CXL resolves two-part references `a.b` first against the input
qualifier namespace, then against the system namespace roots
(`$pipeline.*`, `$window.*`, `$meta.*`, `$doc.*`). Allowing a qualifier
to shadow a reserved root (per RESOLUTION EC-3) or to itself contain a
dot (per RESOLUTION EC-4) would force the resolver to guess between two
valid parses. The check is early (bind_schema) so users see the error
before any expression typechecking runs.
