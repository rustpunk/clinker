# Error E305: Combine where-clause has no cross-input comparisons

## What it means

The `where:` predicate on a combine node contains no comparison that
references fields from two or more inputs. Combine is intended for
correlated streams — without a cross-input predicate, you are asking
for a Cartesian product, which combine deliberately does not support.

## Example

```yaml
# Invalid: "true" — no cross-input correlation
nodes:
  - type: combine
    name: bad
    input:
      orders: orders
      products: products
    where: "true"                         # ERROR: no cross-input comparison
    cxl: |
      emit orders.id, products.name
```

```yaml
# Invalid: predicate only touches one input
nodes:
  - type: combine
    name: bad
    input:
      orders: orders
      products: products
    where: "orders.id > 0"                # ERROR: only touches "orders"
    cxl: |
      emit orders.id
```

```yaml
# Corrected: at least one cross-input comparison
nodes:
  - type: combine
    name: good
    input:
      orders: orders
      products: products
    where: "orders.product_id == products.id and orders.qty > 0"
    cxl: |
      emit orders.id, products.name
```

## How to fix

Add at least one comparison that references fields from two or more
inputs. Usually this is an equality on the correlating key
(`left.fk == right.pk`), but range predicates
(`events.ts between windows.start and windows.end`) are also valid
cross-input forms.

If you truly want an unconditional cross product, that is a different
operator (a Cartesian-product construct is planned but not yet
available) — combine is not it.

## Technical context

Per spec §Combine Semantics, the where-clause is decomposed into
equality conjuncts, range conjuncts, and a residual term. Each
conjunct is classified by the set of input qualifiers it references.
E305 fires when every conjunct touches at most one input — there is
no way for the planner to pair rows from different inputs.

Contrast with W305 (warning): a predicate with only range or residual
cross-input comparisons is legal but slow — W305 suggests adding an
equi-conjunct for performance. E305 is the stronger condition: there
are no cross-input comparisons at all.
