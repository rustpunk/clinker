# Error E362: a malformed `join_values` declaration on an output

## What it means

`join_values` tells a CSV sink how to collapse a `multiple:` field into one
delimited cell — the write-side inverse of a source's `split_values`. This
diagnostic fires when the declaration itself is malformed, before any record is
written:

- **Declared on a non-CSV output.** Only the `csv` writer consumes `join_values`.
  A `json` output already emits a native array, and the `xml` / `fixed_width` /
  segment writers have no delimited-cell join, so declaring the block there is a
  silent no-op — rejected the same way an unread `split_values` block is on a
  source (`E358`).
- **A duplicate `field`.** Two entries for the same field leave it ambiguous
  which delimiter and policy win.
- **An empty `delimiter`.** There would be nothing to separate the joined values.
- **A multi-character `delimiter` or `escape` under `on_conflict: escape`.** The
  escape policy escapes and un-escapes character by character, so a
  multi-character delimiter or escape could not round-trip exactly. Use
  `encode_json` for an arbitrary delimiter, or shorten to one character.

```
output 'report': `join_values` is declared on a `json` output, whose writer
never consumes it — the declaration would be a silent no-op
```

## Example

```yaml
# Rejected: join_values on a JSON output, whose writer emits a native array and
# never reads the block.
  - type: output
    name: report
    input: orders
    config:
      name: report
      type: json
      path: ./out/report.json
      join_values:
        - tags
```

```yaml
# Rejected: `on_conflict: escape` with a two-character delimiter cannot
# round-trip exactly.
      join_values:
        - { field: tags, delimiter: "::", on_conflict: escape }
```

```yaml
# Accepted: a well-formed CSV join. A multi-value field with no entry at all is
# also fine — it joins with the default `;` and `on_conflict: error`.
  - type: output
    name: report
    input: orders
    config:
      name: report
      type: csv
      path: ./out/report.csv
      join_values:
        - tags                              # delimiter ";", on_conflict: error
        - { field: notes, delimiter: "|", on_conflict: escape, escape: "\\" }
```

## How to fix

1. **On a non-CSV output, remove the `join_values` block.** A `json` output emits
   a native array with no configuration; the `xml` / `fixed_width` / segment
   writers have no delimited-cell join.

2. **Declare each field once**, with the delimiter and policy it should join
   with.

3. **Give a non-empty delimiter** (the default is `;`).

4. **Under `on_conflict: escape`, use a single-character `delimiter` and
   `escape`.** For an arbitrary delimiter, switch to `on_conflict: encode_json`,
   which encodes the whole field as a JSON array and has no character-level
   escaping constraint.

## Technical context

`E362` validates only the shape of the declaration, not whether a multi-value
column actually reaches the output. Unlike `split_values` (whose `E358` check
requires the named column to be `multiple: true` on the source schema), a
write-side array can arise at runtime — a `match: collect` combine output, for
instance — so requiring the field to be a statically-declared `multiple:` column
would reject a legitimate override. An entry that never matches an array is
simply an inert override.

A `multiple:` column reaching a CSV output needs no `join_values` entry at all:
the writer joins with the default `;` and `on_conflict: error`. The block is only
for overriding the delimiter or collision policy per field. The default
`on_conflict: error` is what makes a defaulted delimiter safe — a value that
itself contains the delimiter is dead-lettered rather than emitted as a cell that
would split back wrongly downstream.
