# Error E362: a malformed `join_values` declaration on an output

## What it means

`join_values` is the write-side multi-value declaration ‚Äî the inverse of a
source's `split_values`. Two writers consume it: the CSV writer collapses the
values into one delimited cell (`delimiter` / `on_conflict` / `escape`), and the
XML writer emits them as repeated child elements (`repeat_as` / `wrap_in`). This
diagnostic fires when the declaration itself is malformed, before any record is
written:

- **Declared on an output that consumes it neither way.** Only the `csv` and
  `xml` writers read `join_values`. A `json` output already emits a native array,
  and the `fixed_width` / segment writers have no multi-value encoding to
  configure, so declaring the block there is a silent no-op ‚Äî rejected the same
  way an unread `split_values` block is on a source (`E358`).
- **A duplicate `field`.** Two entries for the same field leave its encoding
  ambiguous. This applies to both consumers.

The remaining checks are the CSV writer's ‚Äî the XML writer reads only `repeat_as`
and `wrap_in` (validated as legal XML names when the record is written) and
ignores `delimiter` / `on_conflict` / `escape`, so a defaulted delimiter on an
XML entry is not a fault:

- **An empty `delimiter`.** There would be nothing to separate the joined values.
- **A multi-character `delimiter` under `on_conflict: error` or `escape`.** Under
  `error` the collision check tests whether a value contains the delimiter, which
  misses a spurious boundary a multi-character delimiter can form between two
  adjacent values; under `escape` the char-by-char escaping cannot round-trip a
  multi-character delimiter. `encode_json` ignores the delimiter, so it places no
  constraint on it ‚Äî use it for an arbitrary delimiter.
- **A multi-character `escape`, or an `escape` equal to the `delimiter`, under
  `on_conflict: escape`.** Escaping is per character, and if the escape and
  delimiter were the same character an escaped delimiter and an escape marker
  would be indistinguishable on read.

```
output 'report': `join_values` is declared on a `json` output, whose writer
never consumes it ‚Äî the declaration would be a silent no-op
```

## Example

```yaml
# Rejected: join_values on a JSON output, whose writer emits a native array and
# never reads the block.
  - type: output
    name: report
    input: orders
    config:
      name: report
      type: json
      path: ./out/report.json
      join_values:
        - tags
```

```yaml
# Rejected: `on_conflict: escape` with a two-character delimiter cannot
# round-trip exactly.
      join_values:
        - { field: tags, delimiter: "::", on_conflict: escape }
```

```yaml
# Accepted: a well-formed CSV join. A multi-value field with no entry at all is
# also fine ‚Äî it joins with the default `;` and `on_conflict: error`.
  - type: output
    name: report
    input: orders
    config:
      name: report
      type: csv
      path: ./out/report.csv
      join_values:
        - tags                              # delimiter ";", on_conflict: error
        - { field: notes, delimiter: "|", on_conflict: escape, escape: "\\" }
```

```yaml
# Accepted: an XML join renaming the repeated element and adding a container. A
# multi-value field with no entry emits bare repeats named after the field.
  - type: output
    name: report
    input: orders
    config:
      name: report
      type: xml
      path: ./out/report.xml
      join_values:
        - { field: tags, repeat_as: Tag, wrap_in: Tags }
```

## How to fix

1. **On an output that is neither `csv` nor `xml`, remove the `join_values`
   block.** A `json` output emits a native array with no configuration; the
   `fixed_width` / segment writers have no multi-value encoding to configure.

2. **Declare each field once**, with the encoding it should use.

3. **Give a non-empty delimiter** (the default is `;`).

4. **Under `on_conflict: error` or `escape`, use a single-character
   `delimiter`** ‚Äî and under `escape`, a single-character `escape` that differs
   from the delimiter. For an arbitrary delimiter, switch to
   `on_conflict: encode_json`, which encodes the whole field as a JSON array and
   places no constraint on the delimiter.

## Technical context

`E362` validates only the shape of the declaration, not whether a multi-value
column actually reaches the output. Unlike `split_values` (whose `E358` check
requires the named column to be `multiple: true` on the source schema), a
write-side array can arise at runtime ‚Äî a `match: collect` combine output, for
instance ‚Äî so requiring the field to be a statically-declared `multiple:` column
would reject a legitimate override. An entry that never matches an array is
simply an inert override.

A `multiple:` column reaching a CSV output needs no `join_values` entry at all:
the writer joins with the default `;` and `on_conflict: error`. The block is only
for overriding the delimiter or collision policy per field. The default
`on_conflict: error` is what makes a defaulted delimiter safe ‚Äî a value that
itself contains the delimiter is dead-lettered rather than emitted as a cell that
would split back wrongly downstream.
