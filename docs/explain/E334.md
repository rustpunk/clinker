# Error E334: Spill directory equals the staging directory

## What it means

Your `storage.spill.dir` and `storage.staging.dir` resolve to the **same
directory**. Spill files (the on-disk overflow blocking operators write
when the memory budget tightens) and staged source copies (verbatim copies
of input files) serve different purposes, are sized differently, and are
cleaned up on different schedules — interleaving them in one directory makes
disk-budget accounting, cleanup, and troubleshooting ambiguous. The run is
refused at startup so the two never share space.

The startup diagnostic names the shared path:

```
E334 storage.spill.dir and storage.staging.dir both resolve to
/var/clinker/scratch; spill files and staged source copies must not share a
directory — give them distinct paths. See: clinker explain --code E334
```

## Example

```toml
# clinker.toml — both keys point at the same directory.
[storage.spill]
dir = "/var/clinker/scratch"

[storage.staging]
enabled = true
dir     = "/var/clinker/scratch"
```

```toml
# Corrected: give spill and staging distinct directories.
[storage.spill]
dir = "/var/clinker/spill"

[storage.staging]
enabled = true
dir     = "/var/clinker/staging"
```

## How to fix

1. **Give each its own directory.** Point `storage.spill.dir` and
   `storage.staging.dir` at distinct paths. They may live on the same disk
   (subject to the staging same-device rule for the *sources* — see E333),
   but they must not be the same directory.

## Technical context

E334 is a startup-validation diagnostic raised by `validate_storage_config`
after the plan compiles and before any source-ingest thread spawns. The
equality check pairs the `fs_type` facade's `same_device` probe with a
canonical-path comparison, so two spellings of one directory (a relative
and an absolute path, or a path with a trailing slash) still collide, while
two distinct directories that merely sit on one volume do not.

Validating up front keeps the spill and staging spaces cleanly separated
before either is written.
