# Warning W305: Combine where-clause has no equality conjuncts

## What it means

The combine's `where:` predicate decomposes into range or residual
conjuncts only — no equality conjuncts. The planner cannot use
HashBuildProbe (its fastest equi-strategy) and falls back to
IEJoin, SortMergeJoin, or BlockNestedLoop. These are correct but
much slower on large inputs than hash-based equi-joins.

## Example

```yaml
# Warning: all range conjuncts, no equality
nodes:
  - type: combine
    name: slow
    input:
      events:  events
      windows: windows
    where: "events.ts between windows.start and windows.end"   # range-only
    cxl: |
      emit events.id, windows.label
```

```yaml
# Warning: single range predicate
nodes:
  - type: combine
    name: also_slow
    input:
      orders:   orders
      discounts: discounts
    where: "orders.amount > discounts.min_amount and
            orders.amount <= discounts.max_amount"             # range-only
    cxl: |
      emit orders.id, discounts.code
```

## How to fix

If your domain has a logical equality predicate that can combine
with the range, add it — it will dominate strategy selection:

```yaml
# No warning: equality dominates, range is residual
nodes:
  - type: combine
    name: fast
    input:
      events:  events
      windows: windows
    where: "events.zone == windows.zone and
            events.ts between windows.start and windows.end"
    cxl: |
      emit events.id, windows.label
```

If no equality predicate exists (e.g., generic temporal joins),
W305 is informational — the range-join strategies are still
correct, just slower. In that case, keep input sizes bounded and
consider adding `estimated_rows:` on sources so the planner picks
the right range-join variant (IEJoin vs SortMerge).

## Technical context

Per spec §Predicate Decomposition, the where-clause is split into
`equality_conjuncts`, `range_conjuncts`, and `residual` (see
`DecomposedPredicate` in `plan/combine.rs`). Strategy selection
prefers HashBuildProbe when equality conjuncts exist, falling
through to IEJoin, SortMerge, and BlockNestedLoop as equality
support recedes. W305 fires when `equality_conjuncts.is_empty()`
and at least one `range_conjunct` or residual cross-input predicate
exists. Contrast with E305 (no cross-input predicates at all — not
a warning, an error).

Per RESOLUTION W-18, W305 is the "no equality conjuncts" warning;
the original overloaded W305 was split into W305 (this warning)
and W306 (ambiguous driving input) so each is independently
user-actionable.
