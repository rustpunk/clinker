# Error E361: a source format cannot produce the multi-value column it declares

## What it means

A schema column declared `multiple: true` holds more than one value, and CXL
sees it as an `array`. Only a source format that can actually carry repetition
delivers one. This diagnostic fires when the declaration and the format
disagree, which would otherwise typecheck downstream code against an array and
then hand it the single value the reader really parsed.

It is the read-side half of the rule `E359` enforces on the write side: `E359`
rejects a multi-value column reaching a writer that cannot encode it, and `E361`
rejects one on a source that cannot produce it.

Where each input format stands:

- **`json`** — a JSON array is the format's own shape. `multiple: true` works on
  its own.
- **`xml`** — repeated child elements are collected into one array, in document
  order. `multiple: true` works on its own.
- **`csv` / `fixed_width`** — one value per cell on the wire, but a cell's text
  may hold several separated by a delimiter. The column is accepted once a
  `split_values` entry says what that delimiter is, and rejected before then.
- **`edifact` / `x12` / `hl7` / `swift`** — repetition in these formats is a
  positional coordinate, not a list, and no `split_values` entry changes that.
  See the technical context below.

```
source 'orders': the schema declares the multi-value column(s) 'tags'
(`multiple: true`), and a `csv` source has no way to produce a field holding
more than one value
```

## Example

```yaml
# Rejected: a CSV source declares a multi-value column with nothing to make it.
nodes:
  - type: source
    name: orders
    config:
      name: orders
      type: csv
      path: ./orders.csv
      schema:
        - { name: order_id, type: string }
        - { name: tags, type: string, multiple: true }
```

```yaml
# Corrected: the cell holds `a;b;c`, so say so.
      split_values:
        - field: tags
          delimiter: ";"
      schema:
        - { name: order_id, type: string }
        - { name: tags, type: string, multiple: true }
```

```yaml
# Corrected (each cell really does hold one value): drop the declaration.
      schema:
        - { name: order_id, type: string }
        - { name: tags, type: string }
```

## How to fix

1. **On a `csv` or `fixed_width` source, add the `split_values` entry.** Name
   the input field — the `source_name` when the column aliases a
   differently-named field — and the delimiter its cells actually use.

2. **On a segment format, drop `multiple: true`.** For an HL7 source, declare
   the field under `options.split_fields` with its repetition and component
   counts instead: that produces one column per occurrence, which is the shape
   the wire really has. For `edifact`, `x12`, and `swift` there is no such
   declaration today, and each occurrence arrives verbatim in the element text.

3. **Drop `multiple: true`** if the field never actually repeats.

4. **If the schema is shared across sources** and only some of them repeat the
   field, clear it for this one with a channel source patch
   (`schema: { tags: { multiple: false } }`) rather than editing the shared
   declaration.

## Technical context

`Column::bound_type()` binds a `multiple: true` column as `array` for every
format — the declared `type:` describes each ELEMENT and drives coercion, so the
attribute alone decides the bound type. That is correct, and it is why the
declaration has to be gated: nothing further down the pipeline re-checks it. The
schema-coercion layer's scalar arm coerces and passes a bare value through
rather than fabricating a one-element array, so an ungated declaration produced
no error anywhere — just a scalar where the typechecker had promised an array.

The one-element-array normalization the JSON and XML readers apply is
deliberately not generalized to the other formats. Those two have a genuine
ambiguity — XML cannot distinguish a collection of one from a scalar, so the
declaration is what disambiguates. A CSV cell is always exactly one string, so
the same normalization would wrap `a;b;c` into `["a;b;c"]` and make
`size(tags)` return 1 on every row forever: a loud authoring error converted
into a quiet permanent wrong answer.

The segment formats are marked permanent rather than pending. A repeated
composite serializes as two structural axes interleaved in one element —
`11:B:1^12:B:2` is two repetitions of a three-component composite — which a flat
array of values cannot represent without losing the component axis. The faithful
model is one column per coordinate, which is what the HL7 `split_fields`
declaration already builds and what its writer inverts byte-exactly.
