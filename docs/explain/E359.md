# Error E359: a multi-value column reaches an output that cannot encode it

## What it means

A column declared `multiple: true` — the column holds more than one value —
reaches an output whose format has no encoding for such a field. The writers
reject a multi-value payload rather than degrading it, so without this gate the
run would start, stream some records, and then fail partway through on the first
record whose column actually carried two values.

Two declarations raise it. One is a `multiple: true` column on a SOURCE that
reaches the output through the DAG. The other is `multiple: true` on the
output's OWN `schema:` block: the attribute is direction-neutral by design — it
describes the shape of the data, so a writer that can encode repetition reads
the same declaration — but the remaining writers do not encode it, and the
fixed-width writer's column conversion drops the attribute outright. Declaring it
there would be accepted and quietly ignored.

The `json` output encodes a multi-value field as a JSON array, the `csv` output
joins it into one delimited cell (`join_values`, see the CSV format reference),
and the `xml` output emits it as repeated child elements (`join_values`, see the
XML format reference). The `fixed_width` writer has a well-established convention
for it that is not implemented yet; the segment formats (`edifact`, `x12`, `hl7`,
`swift`) emit a fixed positional grammar with no repetition slot.

```
output 'report': source 'orders' declares the multi-value column(s) 'tags'
(`multiple: true`), and a `fixed_width` output has no encoding for a field
holding more than one value
```

## Example

<!-- explain-test: expect-code=E359 -->
```yaml
# Rejected: a multi-value column flows into a fixed-width output, which has no
# writer for a multi-value field yet.
nodes:
  - type: source
    name: orders
    config:
      name: orders
      type: xml
      path: ./orders.xml
      options:
        record_path: Orders/Order
      schema:
        - { name: order_id, type: string }
        - { name: tags,     type: string, multiple: true }

  - type: output
    name: report
    input: orders
    config:
      name: report
      type: fixed_width
      path: ./out/report.txt
      schema:
        - { name: order_id, type: string, start: 0, width: 10 }
```

```yaml
# Corrected (write CSV): the field joins into one delimited cell, defaulting to
# `;` with `on_conflict: error`. See `join_values` in the CSV format reference.
  - type: output
    name: report
    input: orders
    config:
      name: report
      type: csv
      path: ./out/report.csv
```

```yaml
# Corrected (write JSON): the array is the format's native shape.
  - type: output
    name: report
    input: orders
    config:
      name: report
      type: json
      path: ./out/report.json
```

```yaml
# Corrected (write XML): the values emit as repeated child elements, one per
# value. See `join_values` in the XML format reference to rename them.
  - type: output
    name: report
    input: orders
    config:
      name: report
      type: xml
      path: ./out/report.xml
```

```yaml
# Also rejected: the declaration on the output's own schema, where no writer
# honors it.
  - type: output
    name: report
    input: orders
    config:
      name: report
      type: fixed_width
      path: ./out/report.txt
      schema:
        - { name: order_id, type: string, start: 0, width: 10 }
        - { name: tags, type: string, start: 10, width: 40, multiple: true }
```

```yaml
# Corrected (fan out first): one record per value, so the column that reaches
# the sink holds a single value.
  - type: source
    name: orders
    config:
      name: orders
      type: xml
      path: ./orders.xml
      options:
        record_path: Orders/Order
      split_to_rows:
        - tags
      schema:
        - { name: order_id, type: string }
        - { name: tags,     type: string }
```

## How to fix

1. **Write the stream to a `json`, `csv`, or `xml` output.** JSON emits a native
   array; CSV joins the values into one delimited cell (`join_values`, defaulting
   to `;` with `on_conflict: error`); XML emits them as repeated child elements
   (`join_values` `repeat_as` / `wrap_in` rename them).

2. **Fan the values out to rows** with `split_to_rows` on the source, so each
   output record carries one value and the column is no longer multi-value.

3. **Collapse the column to a single value** in a transform before the sink.

4. **Drop `multiple: true`** from the source column if the field never actually
   repeats — the declaration, not the data, is what this gate reads. On an
   output's own `schema:`, drop it outright: there is no writer to honor it.

5. **If the schema is shared across sources** and only some of them repeat the
   field, clear it for this one with a channel source patch rather than editing
   the shared declaration:

   ```yaml
   sources:
     orders:
       schema:
         tags: { multiple: false }
   ```

## Technical context

Reachability is computed source-to-output over the DAG, unioning every input of
every node — a combine can project a column off its reference side, not only its
driver — and iterating to a fixpoint, so a node declared before its producer is
reached like any other. It is deliberately conservative: a column dropped by a
transform before the sink still raises the error. A false positive is a
compile-time message the author can read and resolve; a false negative is a run
that dies partway through with output already written.

Which formats qualify lives in one per-format capability table, so a writer
gaining multi-value support relaxes exactly one entry and nothing else in the
gate changes. The read side has its own table and its own diagnostic (`E361`);
the two are the same rule pointed in opposite directions, and neither consults
the other. `json`, `csv`, and `xml` encode a multi-value field today; the
remaining writer-side support is tracked per format:
[fixed-width](https://github.com/rustpunk/clinker/issues/918).
