# Error E359: a multi-value column reaches an output that cannot encode it

## What it means

A source declares a column `multiple: true` — the column holds more than one
value — and a downstream output writes a format with no encoding for such a
field. The writers reject a multi-value payload rather than degrading it, so
without this gate the run would start, stream some records, and then fail
partway through on the first record whose column actually carried two values.

Today only the `json` output can encode a multi-value field, as a JSON array.
The `csv`, `xml`, and `fixed_width` writers each have a well-established
convention for it that is not implemented yet; the segment formats (`edifact`,
`x12`, `hl7`, `swift`) emit a fixed positional grammar with no repetition slot.

```
output 'report': source 'orders' declares the multi-value column(s) 'tags'
(`multiple: true`), and a `csv` output has no encoding for a field holding more
than one value
```

## Example

```yaml
# Rejected: a multi-value column flows into a CSV output.
nodes:
  - type: source
    name: orders
    config:
      name: orders
      type: xml
      path: ./orders.xml
      options:
        record_path: Orders/Order
      schema:
        - { name: order_id, type: string }
        - { name: tags,     type: string, multiple: true }

  - type: output
    name: report
    input: orders
    config:
      name: report
      type: csv
      path: ./out/report.csv
```

```yaml
# Corrected (write JSON): the array is the format's native shape.
  - type: output
    name: report
    input: orders
    config:
      name: report
      type: json
      path: ./out/report.json
```

```yaml
# Corrected (fan out first): one record per value, so the column that reaches
# the sink holds a single value.
  - type: source
    name: orders
    config:
      name: orders
      type: xml
      path: ./orders.xml
      options:
        record_path: Orders/Order
      split_to_rows:
        - tags
      schema:
        - { name: order_id, type: string }
        - { name: tags,     type: string }
```

## How to fix

1. **Write the stream to a `json` output**, where a multi-value field is an
   ordinary array.

2. **Fan the values out to rows** with `split_to_rows` on the source, so each
   output record carries one value and the column is no longer multi-value.

3. **Collapse the column to a single value** in a transform before the sink.

4. **Drop `multiple: true`** from the source column if the field never actually
   repeats — the declaration, not the data, is what this gate reads.

5. **If the schema is shared across sources** and only some of them repeat the
   field, clear it for this one with a channel source patch rather than editing
   the shared declaration:

   ```yaml
   sources:
     orders:
       schema:
         tags: { multiple: false }
   ```

## Technical context

Reachability is computed source-to-output over the compiled DAG, the same walk
that attributes `$doc.*` paths to the sources that deliver them. It is
deliberately conservative: a column dropped by a transform before the sink still
raises the error. A false positive is a compile-time message the author can read
and resolve; a false negative is a run that dies partway through with output
already written.

Which formats qualify lives in one per-format capability table, so a writer
gaining multi-value support relaxes exactly one entry and nothing else in the
gate changes. The read side has its own table and its own diagnostic (`E361`);
the two are the same rule pointed in opposite directions, and neither consults
the other. Writer-side support is tracked per format:
[XML](https://github.com/rustpunk/clinker/issues/916),
[CSV](https://github.com/rustpunk/clinker/issues/917), and
[fixed-width](https://github.com/rustpunk/clinker/issues/918).
