# Error E331: Spill directory on a network filesystem

## What it means

The `storage.spill.dir` you configured resolves to a **network or
userspace-bridged filesystem** — NFS, SMB / CIFS, or FUSE. Spilling onto a
network share reintroduces exactly the fragility that disk spill exists to
avoid: a soft-mounted NFS share can silently truncate a write, mmap-backed
access can lose data on a transient disconnect, and every spill round-trip
pays network latency. The run is refused at startup rather than allowed to
spill onto an unreliable transport.

The startup diagnostic names the offending setting:

```
E331 storage.spill.dir /mnt/nfs/clinker-spill is on a network filesystem
(NFS/SMB/CIFS/FUSE); a spill target on a soft-mounted share risks silent
truncation and data loss — point storage.spill.dir at a local disk.
See: clinker explain --code E331
```

## Example

```toml
# clinker.toml — spilling onto the same NFS share the inputs live on.
[storage.spill]
dir = "/mnt/nfs/clinker-spill"
```

```toml
# Corrected: spill to a local disk; keep network shares for inputs only.
[storage.spill]
dir = "/var/clinker/spill"
```

## How to fix

1. **Point `storage.spill.dir` at a local disk.** Choose a path on a
   directly-attached block device with room for the spill footprint.
   `df -T <path>` should report a local filesystem (ext4 / xfs / btrfs /
   apfs / NTFS), not `nfs`, `cifs`, or a FUSE type.

2. **If you must read inputs from a network share,** stage them to local
   disk first with `[storage.staging]` — that is what staging is for — and
   keep the spill volume local and separate.

## Technical context

E331 is a startup-validation diagnostic raised by `validate_storage_config`
after the plan compiles and before any source-ingest thread spawns. The
filesystem class is read through the cross-platform `fs_type` facade: Linux
matches the `statfs` `f_type` magic against NFS / SMB / SMB2 / CIFS / FUSE,
macOS matches the `f_fstypename` string (`nfs`, `smbfs`, `cifs`, `webdav`,
`afpfs`, or any `fuse`-containing name), and Windows maps `GetDriveTypeW`
`DRIVE_REMOTE`.

Validating up front fails a misconfigured volume while the run is still
cheap to abandon, rather than after minutes of work when the first spill
hits the flaky transport.
