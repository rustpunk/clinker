# Error E302: Qualified field collision across combine inputs

## What it means

After schema merge, the same fully-qualified name `<qualifier>.<field>`
appears in more than one input schema. This is almost always caused by
the same qualifier being used for two different inputs (which is already
rejected as a YAML map duplicate), or by an input's schema being merged
twice through an upstream alias.

## Example

```yaml
# Invalid: "orders" used twice through different routes
nodes:
  - type: source
    name: orders_raw
    config: { type: csv, path: orders.csv, schema: [...] }

  - type: transform
    name: orders_clean
    input: orders_raw
    config:
      cxl: |
        emit *

  - type: combine
    name: bad
    input:
      orders: orders_raw         # same qualifier "orders"
      orders: orders_clean        # ERROR: YAML duplicate, or after
                                  # indexmap, last wins + E302 on merge
    where: "orders.id == orders.id"
    cxl: |
      emit orders.id
```

```yaml
# Corrected: distinct qualifiers per input
nodes:
  - type: combine
    name: good
    input:
      raw: orders_raw
      clean: orders_clean
    where: "raw.id == clean.id"
    cxl: |
      emit raw.id, clean.normalized_name
```

## How to fix

Give each input a distinct qualifier. If you need to pull two versions
of the same stream (pre- and post-transform, say), give each an
intention-revealing qualifier (`raw`, `clean`, `enriched`, etc.).

## Technical context

Combine builds its merged-row schema by taking the disjoint union of
each input's schema keyed by qualifier. Per RESOLUTION EC-5, there are
no implicit "unqualified" fields — `orders.id` and `products.id` are
distinct names, not a collision. E302 fires only when two inputs
actually land on the same `qualifier.field` string, which requires
duplicate qualifiers. This keeps the error space tight: users can't
accidentally trigger E302 just by having same-named fields in
different inputs.
