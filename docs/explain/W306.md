# Warning W306: Combine planner cannot determine optimal driving input

## What it means

The planner tried to pick the best driving input (build side for
HashBuildProbe, outer for nested-loop) but the cardinality
estimates for every input are either tied or entirely absent. The
planner fell back to a deterministic tiebreaker (first declared
input in the `input:` map), which may or may not be optimal.

## Example

```yaml
# Warning: neither source has estimated_rows; tie by default
nodes:
  - type: source
    name: orders
    config: { type: csv, path: orders.csv, schema: [...] }
                                          # no estimated_rows
  - type: source
    name: products
    config: { type: csv, path: products.csv, schema: [...] }
                                          # no estimated_rows

  - type: combine
    name: ambiguous
    input:
      orders: orders
      products: products
    where: "orders.product_id == products.id"
    cxl: |
      emit orders.id, products.name
```

## How to fix

Pick whichever applies:

1. **Add `estimated_rows:` to your sources** so the planner has
   cardinality data:
   ```yaml
   - type: source
     name: products
     estimated_rows: 500
     config: { type: csv, path: products.csv, schema: [...] }
   ```

2. **Specify `drive:` explicitly** on the combine node:
   ```yaml
   - type: combine
     name: enrich
     input: { orders: orders, products: products }
     drive: "products"                    # products is smaller
     where: "orders.product_id == products.id"
     cxl: |
       emit orders.id, products.name
   ```

3. **Ignore the warning** if the pipeline performance is acceptable.
   The tiebreak choice is deterministic, so runs are reproducible
   even without the hint.

## Technical context

Driving-input selection drives which input is hashed (build side)
and which streams (probe side) in HashBuildProbe, and which is the
outer vs inner loop in nested-loop strategies. Incorrect selection
can swing runtime by orders of magnitude for skewed cardinalities.
When the planner has no cardinality signal, any choice is a guess —
W306 surfaces the guess so users can confirm or override.

Per RESOLUTION W-18, W306 is distinct from W305 (no equality
conjuncts) because the two remediations are different: W305 asks for
a predicate change; W306 asks for a cardinality hint or a `drive:`
field. Splitting them lets each warning carry one clear action.
