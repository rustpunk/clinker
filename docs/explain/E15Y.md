# Error E15Y: relaxed_correlation_key incompatible with strategy: streaming

## What it means

An Aggregate node opted into `relaxed_correlation_key: true` — relaxing
the strict `group_by ⊇ correlation_key` requirement (`E151`) so that one
correlation group can span multiple aggregate groups — but its
`strategy:` is `streaming`. Streaming aggregates emit results at every
group-boundary close, which happens *before* the terminal correlation
commit. The relaxed opt-in exists so that records dropped at
`CorrelationCommit` can retroactively retract or recompute their
contributions to surviving aggregate groups; once a streaming aggregate
has flushed a group's row, that rollback window is closed.

## Example

```yaml
error_handling:
  correlation_key: order_id

nodes:
  - type: aggregate
    name: dept_totals
    input: validate
    config:
      group_by: [department]
      strategy: streaming         # E15Y -- incompatible with the opt-in below
      relaxed_correlation_key: true
      cxl: |
        emit total = sum(amount)
```

## How to fix

Either drop the `strategy: streaming` hint (the `hash` default keeps
results buffered until `CorrelationCommit`, which is what the relaxed
path needs), or remove `relaxed_correlation_key: true` and stay on the
strict-superset path.

```yaml
- type: aggregate
  name: dept_totals
  input: validate
  config:
    group_by: [department]
    relaxed_correlation_key: true   # OK -- defaults to hash strategy
    cxl: |
      emit total = sum(amount)
```

## Technical context

Today's `CorrelationCommit` flushes per-group buffers at end-of-DAG and
DLQs entire correlation groups when any contributing record fails
upstream. Hash aggregation buffers its accumulator state until
`finalize` and so coexists with the rollback window. Streaming
aggregation emits per group-boundary close, which under
`relaxed_correlation_key` would deliver an aggregate row to the writer
before the engine knows whether any of its contributing records will be
DLQ'd at commit time.
