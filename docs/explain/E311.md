# Error E311: Combine `match: collect` has a non-empty `cxl:` body

## What it means

A combine node with `match: collect` declared a non-empty `cxl:` body.
Under `collect` the output row is auto-derived: the driver's fields
plus one new field — named after the build-side qualifier — whose
value is an array of all matched build-side records. The body would
never be evaluated, so the planner rejects it at compile time rather
than silently drop it.

## Example

```yaml
# Invalid: match: collect with a non-empty cxl body
nodes:
  - type: combine
    name: bad
    input:
      orders: orders
      line_items: line_items
    where: "orders.id == line_items.order_id"
    match: collect
    cxl: |
      emit orders.id, line_items.sku       # ERROR: body is never evaluated under collect
```

```yaml
# Corrected: drop the body — the output row is auto-derived
nodes:
  - type: combine
    name: good
    input:
      orders: orders
      line_items: line_items
    where: "orders.id == line_items.order_id"
    match: collect
    # Output per driving row: { ...orders fields, line_items: [ {...line_item}, ... ] }
```

If custom projection over the collected array is needed, place a
`transform` node downstream of the combine and operate on the array
field there.

## How to fix

Either remove the `cxl:` body or change `match:` to `first` / `all`.

- Use `match: collect` when all matches should be grouped into a single
  output row per driver.
- Use `match: first` for scalar enrichment (at most one match per
  driver) with custom projection via `cxl:`.
- Use `match: all` for fan-out (one output row per match) with custom
  projection via `cxl:`.

## Technical context

Under `match: collect`, the combine executor synthesizes the output
record directly: it copies the driver's fields and attaches a new
field, named after the build-side qualifier, whose value is a
`Value::Array` containing one `Value::Map` per matched build record.
A driver with zero matches produces an empty array, not `null`. No
`ProgramEvaluator` runs for the body, so there is no place for body
semantics to honour. Rejecting non-empty bodies at compile time
avoids the class of bug where a user writes projection logic
expecting it to run, only to see their query silently ignored.
