# Error E360: a source declares the retired `array_paths:` key

## What it means

`array_paths:` was the earlier way to say "this field repeats, fan it out" on a
JSON or XML source. It has been replaced by three declarations that each say one
thing: `split_to_rows:` fans a repeated field out to one record per occurrence,
`split_values:` parses a delimited cell into several values, and `multiple: true`
on a schema column keeps the repeats as one array.

Nothing reads `array_paths:` any more. A source's config block is merged into its
node, and YAML merging accepts keys the engine does not name, so an untouched
`array_paths:` block would deserialize to nothing and the pipeline would compile
and run — emitting one record per document where it used to emit one per array
element, with every downstream count, join, and aggregate quietly wrong. This
diagnostic exists so that migration fails at compile instead.

```
source 'orders' declares `array_paths:`, which the multi-value declarations
replaced — the key is no longer read, so leaving it in place would drop the
fan-out silently
```

## Example

```yaml
# Rejected: the retired key is still present.
nodes:
  - type: source
    name: orders
    config:
      name: orders
      type: json
      path: ./orders.json
      array_paths:
        - path: line_items
          mode: explode
      schema:
        - { name: order_id, type: string }
        - { name: sku, type: string }
        - { name: qty, type: int }
```

```yaml
# Corrected: the same fan-out as a `split_to_rows` entry.
      split_to_rows:
        - field: line_items
          mode: extract
      schema:
        - { name: order_id, type: string }
        - { name: sku, type: string }
        - { name: qty, type: int }
```

## How to fix

Translate each `array_paths` entry and delete the block:

1. **An `explode` path becomes a `split_to_rows:` entry.** `mode: extract` (the
   default) reproduces the old projection — the element's own fields are lifted
   to the top level of each output record. `mode: split` keeps the record shape
   instead, leaving the element's fields under their dotted path.

2. **A path whose values you want kept together becomes `multiple: true`** on
   the schema column that reads it. The occurrences collect into one array on
   one record rather than fanning out.

3. **A delimited cell becomes a `split_values:` entry**, naming the input field
   and its separator, with `multiple: true` on the column that receives the
   parsed values.

If a fan-out declaration was there only to reach a nested field, note that
`split_to_rows` names the repeated element by its record-relative dotted path
(`line_items`, `Items.Item`), the same form the source's flattened field names
use.

## Technical context

The check cannot be a serde `deny_unknown_fields` rejection. The source's format
and discovery options are embedded in the pipeline node with a flattened merge,
which is incompatible with rejecting unknown keys — serde cannot tell a key
meant for the outer struct from one meant for the inner one. So the key is
still parsed, as an opaque value read by nothing but this gate.

That makes the diagnostic specific where a generic unknown-key rejection would
not be: it can name the replacement for each old form rather than reporting an
unrecognized key and leaving the migration to be looked up.
