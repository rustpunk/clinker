# Error E330: Spill directory on an in-memory filesystem

## What it means

The `storage.spill.dir` you configured resolves to an **in-memory
filesystem** — Linux tmpfs or ramfs, or a Windows RAM disk. Blocking
operators spill to disk precisely to move bytes *off* RAM when the memory
budget tightens; a spill target backed by RAM keeps those bytes resident,
so spilling there trades process RSS for page-cache pressure without
freeing any physical memory. The run is refused at startup rather than
allowed to spill uselessly.

On many systemd-managed Linux hosts `/tmp` is mounted as tmpfs, so a
`storage.spill.dir` pointed at a path under `/tmp` (or left at the default
when `$TMPDIR` is tmpfs) commonly trips this.

The startup diagnostic names the offending setting:

```
E330 storage.spill.dir /tmp/clinker-spill is on an in-memory filesystem
(tmpfs/ramdisk); spilling there keeps the bytes in RAM and defeats the
memory budget — point storage.spill.dir at a path on a real block device.
See: clinker explain --code E330
```

## Example

```toml
# clinker.toml — /tmp is tmpfs on this host, so spill bytes never leave RAM.
[storage.spill]
dir = "/tmp/clinker-spill"
```

```toml
# Corrected: point spill at a path on a real block device.
[storage.spill]
dir = "/var/clinker/spill"
```

## How to fix

1. **Point `storage.spill.dir` at a real disk.** Pick a path on a block
   device (ext4 / xfs / btrfs / apfs / NTFS) with room for the spill
   footprint — `df -T <path>` reporting anything other than `tmpfs` /
   `ramfs` is what you want.

2. **Check what backs `/tmp`.** If you relied on the default spill
   location, run `df -T /tmp`. A `tmpfs` line means spill was never moving
   pressure off RAM; set `storage.spill.dir` explicitly.

## Technical context

E330 is a startup-validation diagnostic raised by `validate_storage_config`
after the plan compiles and before any source-ingest thread spawns. The
filesystem class is read through the cross-platform `fs_type` facade: Linux
matches the `statfs` `f_type` magic against tmpfs / ramfs, and Windows maps
`GetDriveTypeW` `DRIVE_RAMDISK`. macOS has no native tmpfs, so a RAM disk
there presents as hfs/apfs over a synthetic device and reads as a local
disk — E330 only ever fires on Linux and Windows.

Validating up front — rather than at the first spill — fails a
misconfigured volume while the run is still cheap to abandon, the lazy trap
DuckDB fell into when its temp-directory setting was honored only at the
first spill (duckdb/duckdb#9401).
