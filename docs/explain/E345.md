# Error E345: unknown record-type discriminator in a multi-record flat file

## What it means

A multi-record CSV or fixed-width source streams one record per line and uses a
**discriminator** — a byte range (fixed-width) or a named column (CSV) — to
decide which declared record type each line belongs to. E345 fires when a line's
discriminator value matches none of the `records:` entries the source declares.

A genuinely undeclared discriminator value is a malformed file or a missing
record-type declaration. By default the run aborts at the offending line so the
bad input is never silently dropped. Under `dlq_granularity: document` the
unmatched line is treated as an envelope-integrity failure that condemns the
whole file to the dead-letter sink, and the run continues with the remaining
files.

```
E345 line 3: unknown record-type discriminator "X" — no `records:` entry
declares this tag. Declare a record type with `tag: X`, or set
`dlq_granularity: document` to dead-letter the file.
```

## Example

```yaml
# A payment file with header (H), detail (D), and trailer (T) record types.
# A line whose first byte is `X` matches no declared type → E345.
nodes:
  - type: source
    name: payments
    config:
      name: payments
      type: fixed_width
      glob: ./*.txt
      schema:
        discriminator: { start: 0, width: 1 }
        records:
          - { id: header,  tag: H, columns: [ { name: batch_id, type: string, start: 1, width: 9 } ] }
          - { id: detail,  tag: D, columns: [ { name: amount,   type: int, start: 1, width: 9 } ] }
          - { id: trailer, tag: T, columns: [ { name: count,    type: int, start: 1, width: 9 } ] }
```

```text
HBATCH0001
D000000100
Y000000999      <- discriminator "Y" matches no record type → E345
T000000001
```

## How to fix

1. **Declare the missing record type.** Add a `records:` entry whose `tag`
   equals the unmatched discriminator value, with the fields that type carries.

2. **Correct the file** when the value is a corruption or a stray line — the
   discriminator must sit at the configured byte range / column on every line.

3. **Dead-letter the file** by setting `dlq_granularity: document` (with a
   `continue` error strategy) on the source, so a file carrying an unexpected
   discriminator value is condemned to the dead-letter sink and the run
   continues with the other files instead of aborting.

## Technical context

The multi-record reader builds a `tag → record type` map at startup from the
source's `records:` list and a static superset schema whose lead `record_type`
column carries the matched type's id. For each physical line it extracts the
discriminator value, looks up the type, and streams one record on the superset
schema — never buffering the file. A lookup miss is E345, classed as a
structural-validation failure — distinguishable from a trailer **count**
mismatch, but sharing its document dead-letter disposition: by default it
aborts the run at the line; under `dlq_granularity: document` it condemns the
whole file — every already-streamed record of that file is rejected with it —
and the run continues. The physical line number in the message points at the
offending line, counting every line including any header, trailer, or skipped
blank.
