# Error E345: unknown record-type discriminator in a multi-record flat file

## What it means

A multi-record CSV or fixed-width source streams one record per line and uses a
**discriminator** — a byte range (fixed-width) or a named column (CSV) — to
decide which declared record type each line belongs to. E345 fires when a line's
discriminator value matches none of the `records:` entries the source declares.

A genuinely undeclared discriminator value is a malformed file or a missing
record-type declaration. Under the default fail-fast posture the run aborts at
the offending line so the bad input is never silently dropped; under a
record-grained dead-letter strategy the unmatched line is routed to the
dead-letter sink instead of aborting the run.

```
E345 unknown record-type discriminator "X": no `records:` entry declares this
tag. Declare a record type with `tag: X`, or set a non-fail-fast DLQ strategy to
dead-letter unmatched rows.
```

## Example

```yaml
# A payment file with header (H), detail (D), and trailer (T) record types.
# A line whose first byte is `X` matches no declared type → E345.
nodes:
  - type: source
    name: payments
    config:
      name: payments
      type: fixed_width
      glob: ./*.txt
      format_schema:
        discriminator: { start: 0, width: 1 }
        records:
          - { id: header,  tag: H, fields: [ { name: batch_id, type: string, start: 1, width: 9 } ] }
          - { id: detail,  tag: D, fields: [ { name: amount,   type: integer, start: 1, width: 9 } ] }
          - { id: trailer, tag: T, fields: [ { name: count,    type: integer, start: 1, width: 9 } ] }
```

```text
HBATCH0001
D000000100
Y000000999      <- discriminator "Y" matches no record type → E345
T000000001
```

## How to fix

1. **Declare the missing record type.** Add a `records:` entry whose `tag`
   equals the unmatched discriminator value, with the fields that type carries.

2. **Correct the file** when the value is a corruption or a stray line — the
   discriminator must sit at the configured byte range / column on every line.

3. **Dead-letter unmatched rows** by configuring a record-grained dead-letter
   strategy on the source, so an unexpected discriminator value routes the
   single line to the dead-letter sink instead of aborting the run.

## Technical context

The multi-record reader builds a `tag → record type` map at startup from the
source's `records:` list and a static superset schema whose lead `record_type`
column carries the matched type's id. For each physical line it extracts the
discriminator value, looks up the type, and streams one record on the superset
schema — never buffering the file. A lookup miss is E345: fail-fast raises it as
a run-aborting error at the line; a record-grained dead-letter strategy
reclassifies it to a per-record dead-letter so the rest of the file still
streams. This is distinct from a trailer **count** mismatch (a structural-count
failure that condemns the whole document), because an unknown tag is a per-line
classification failure, not an envelope-integrity failure.
