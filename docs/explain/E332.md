# Error E332: Staging directory on a network filesystem

## What it means

The `storage.staging.dir` you configured resolves to a **network or
userspace-bridged filesystem** — NFS, SMB / CIFS, or FUSE. Source staging
exists to copy inputs *off* a flaky network share onto stable local disk
before the run reads them; a staging directory that is itself on a network
share reintroduces the very fragility staging is meant to escape (silent
soft-mount truncation, mmap data loss, latency). The run is refused at
startup rather than allowed to stage onto an unreliable target.

The startup diagnostic names the offending setting:

```
E332 storage.staging.dir /mnt/nfs/staging is on a network filesystem
(NFS/SMB/CIFS/FUSE); staging onto a network share reintroduces the
fragility staging exists to escape — point storage.staging.dir at a local
disk. See: clinker explain --code E332
```

## Example

```toml
# clinker.toml — staging onto a network share, which defeats the purpose.
[storage.staging]
enabled  = true
dir      = "/mnt/nfs/staging"
patterns = ["/mnt/nfs/data/**"]
```

```toml
# Corrected: stage onto a local disk on a different volume than the inputs.
[storage.staging]
enabled  = true
dir      = "/var/clinker/staging"
patterns = ["/mnt/nfs/data/**"]
```

## How to fix

1. **Point `storage.staging.dir` at a local disk.** The whole point of
   staging is to land a stable local copy; pick a path on a
   directly-attached block device. `df -T <path>` should report a local
   filesystem, not `nfs`, `cifs`, or a FUSE type.

2. **Keep staging and source on different volumes.** The staging target
   must also sit on a different physical device than the matched sources
   (see E333), so a local disk separate from the input mount is the right
   choice.

## Technical context

E332 is a startup-validation diagnostic raised by `validate_storage_config`
after the plan compiles and before any source-ingest thread spawns. The
filesystem class is read through the same cross-platform `fs_type` facade
the spill checks use, so the NFS / SMB / CIFS / FUSE detection is identical
on Linux, macOS, and Windows.

Validating up front fails a misconfigured staging volume while the run is
still cheap to abandon, rather than at the first staged copy.
