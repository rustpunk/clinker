# Error E356: envelope declared on a plain single-schema CSV / fixed-width source

## What it means

A plain (single-schema) CSV or fixed-width source declares an `envelope:` block.
A plain flat file is a flat sequence of body records with no header/trailer
structure to pre-scan, so there are no envelope sections to extract. The declared
sections would never be populated, and every `$doc.<section>.<field>` read against
the source would silently resolve to `null` at run time.

This is the same inert-declaration failure the REST-source check (`E349`) prevents:
both declare an `envelope:` on a source that buffers no document. Rejecting it at
compile keeps the fail-fast posture rather than accepting a declaration that can
never do anything.

Envelope extraction on a flat file is a **multi-record** feature. A source that
declares a `discriminator:` and a `records:` list can expose one of its header
record types as a `$doc` section through the `record_type` extract — that source
is **not** rejected. This diagnostic fires only when the schema is a plain
single-record column list.

```
the csv source 'payments' declares an `envelope:` block, but a plain single-schema
csv source carries no header/trailer structure to extract — the declared sections
are inert and every `$doc.<section>.<field>` against this source resolves to null
```

## Example

<!-- explain-test: expect-code=E356 -->
```yaml
# Rejected: a plain single-schema CSV source declares envelope sections.
nodes:
  - type: source
    name: payments
    config:
      name: payments
      type: csv
      path: ./payments.csv
      envelope:
        sections:
          Head:
            extract: { record_type: H }
            fields:
              batch_id: string
      schema:
        - { name: amount, type: int }
```

```yaml
# Corrected (drop the envelope): a plain CSV carries no document sections.
  - type: source
    name: payments
    config:
      name: payments
      type: csv
      path: ./payments.csv
      schema:
        - { name: amount, type: int }
```

```yaml
# Corrected (multi-record): declare a discriminator + record types, and the
# header record type is extractable as a $doc section.
  - type: source
    name: payments
    config:
      name: payments
      type: csv
      path: ./payments.csv
      envelope:
        sections:
          Head:
            extract: { record_type: H }
            fields:
              batch_id: string
      schema:
        discriminator: { start: 0, width: 1 }
        records:
          - { id: header, tag: H, columns: [ { name: batch_id, type: string } ] }
          - { id: detail, tag: D, columns: [ { name: amount,   type: int } ] }
```

## How to fix

1. **Remove the `envelope:` block** from the plain source. A flat, single-schema
   file has no header/trailer sections to extract.

2. **Declare a multi-record schema** if the file genuinely carries header/trailer
   records — a `discriminator:` plus a `records:` list — and expose the header
   record type through the `record_type` extract. See the
   [multi-record](../formats/csv.md#multi-record-files-header--trailer--body)
   reference.

## Technical context

The CSV and fixed-width readers use the default
`FormatReader::prepare_document`, which returns an empty section map — a plain
flat file has no document to pre-scan. Before this check, declaring `envelope:` on
a plain source compiled and resolved to `null` at run time with no signal. The
planner now scans every file-transport source for a CSV / fixed-width format
carrying an `envelope:` block whose schema is a plain single-record column list,
and rejects it. A multi-record schema (`discriminator:` + `records:`) is exempt —
its header record type is extractable — and a REST source is handled by `E349`
instead, so the two rejections never overlap. The multi-record `$doc` path
validation lives separately (see the multi-record reference); this gate resolves
only the plain single-schema case that validation excludes.
