# Error E303: Combine where-clause is not boolean

## What it means

The CXL expression in a combine node's `where:` field does not typecheck
to `Bool`. Combine's where-clause filters the cross-product of input
rows, so it must return a boolean — anything else leaves the planner
with no decision procedure for whether a pair matches.

## Example

```yaml
# Invalid: where evaluates to String, not Bool
nodes:
  - type: combine
    name: bad
    input:
      orders: orders
      products: products
    where: "orders.product_id"         # ERROR: String, not Bool
    cxl: |
      emit orders.id, products.name
```

```yaml
# Invalid: where returns an integer
nodes:
  - type: combine
    name: bad
    input:
      orders: orders
      products: products
    where: "orders.qty + products.price"   # ERROR: Number, not Bool
    cxl: |
      emit orders.id
```

```yaml
# Corrected: explicit boolean predicate
nodes:
  - type: combine
    name: good
    input:
      orders: orders
      products: products
    where: "orders.product_id == products.id"
    cxl: |
      emit orders.id, products.name
```

## How to fix

Wrap the expression so it returns a boolean. Most combine predicates
are equality (`==`), comparison (`<`, `<=`, `>`, `>=`), or combinations
joined with `and` / `or`. If you wrote a bare field reference, you
probably meant to compare it to a field on another input.

## Technical context

The where-clause is typechecked against the merged schema (union of all
input schemas keyed by qualifier) during compile time, before the
planner decomposes it into equality conjuncts and residual terms. A
non-boolean result is rejected here rather than letting the planner
produce a nonsensical strategy. Per spec §Error Handling Strategy, the
diagnostic's primary span covers the `where:` value and the secondary
span points at the resolved top-level expression type.
