# Error E312: Combine has more than 2 inputs

## What it means

The combine node declares three or more inputs in its `input:` map.
The current combine executor supports **binary** (exactly 2-input)
combines only. Multi-way combines must be expressed as a chain of
binary combines until N-ary decomposition is implemented.

## Example

```yaml
# Invalid: three inputs in a single combine
nodes:
  - type: combine
    name: three_way
    input:
      orders: orders
      products: products
      customers: customers         # ERROR: third input not supported
    where: |
      orders.product_id == products.id and
      orders.customer_id == customers.id
    cxl: |
      emit orders.id, products.name, customers.email
```

```yaml
# Corrected: chain two binary combines
nodes:
  - type: combine
    name: enrich_products
    input:
      orders: orders
      products: products
    where: "orders.product_id == products.id"
    cxl: |
      emit orders.*, products.name

  - type: combine
    name: enrich_customers
    input:
      enrich_products: enrich_products
      customers: customers
    where: "enrich_products.customer_id == customers.id"
    cxl: |
      emit enrich_products.*, customers.email
```

## How to fix

Split the combine into a pipeline of binary combines. For each
additional input, add one combine node whose driving input is the
output of the previous combine and whose build input is the new
source. When cardinality information is available, place the smaller
input on the build side (or pin it explicitly with `drive:`) to keep
memory bounded.

## Technical context

The HashBuildProbe strategy builds a hash table on one input (the
build side) and streams the other input (the probe side) against it.
Extending to N>2 requires either a left-deep chain of hash operators
— which the corrected example above expresses explicitly — or a
multi-input algorithm. Chaining keeps the executor simple and lets
the planner reason about per-combine memory budgets independently.
Explicit chaining also makes the join order visible in the pipeline
YAML, which is easier to audit than a planner-chosen order.
