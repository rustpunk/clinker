# Error E313: Combine has no equality conjuncts

## What it means

The combine node's `where:` predicate contains no equality conjuncts
of the form `left_input.field == right_input.field`. The current
combine strategy (HashBuildProbe) requires at least one equality
conjunct — it hashes the build side by those keys and probes from the
driving side. A pure-range predicate (e.g. `pay >= rates.min_pay and
pay <= rates.max_pay`) or a pure arbitrary-CXL predicate has no
hashable key, so no strategy can execute it.

## Example

```yaml
# Invalid: only range conditions, no equality
nodes:
  - type: combine
    name: pay_band
    input:
      employees: employees
      rates: rates
    where: |
      employees.pay >= rates.min_pay and
      employees.pay <= rates.max_pay
    cxl: |
      emit employees.id, rates.band
```

```yaml
# Corrected: add an equality conjunct (a natural partition key)
nodes:
  - type: combine
    name: pay_band
    input:
      employees: employees
      rates: rates
    where: |
      employees.grade == rates.grade and
      employees.pay >= rates.min_pay and
      employees.pay <= rates.max_pay
    cxl: |
      emit employees.id, rates.band
```

## How to fix

Pick whichever applies:

1. **Add an equality conjunct** to the predicate. Most real-world
   range joins carry a natural partition key (grade, region, tenant)
   alongside the range. Adding it lets HashBuildProbe hash by the
   equality key and filter the matches with the range afterwards.

2. **Pre-filter** one side with a `transform` node so the remaining
   combine is small enough that a materialized cross-product is
   acceptable. This is a manual workaround; reserve it for small
   reference tables.

3. **Split the join** into a small exact-match combine plus a
   downstream range filter, if the data shape allows it.

Range-only and sorted-merge strategies (IEJoin, SortMerge) are
planned additions. Until those land, every combine must carry at
least one equality conjunct.

## Technical context

HashBuildProbe hashes the build side by a composite of its equality
key columns, then probes from the driving side by the same composite.
Range predicates cannot partition the build side into a hash: two
records with different values can both satisfy `x >= y`, so a hash
lookup would miss them. Strategies that handle pure-range efficiently
— IEJoin uses interleaved sorted indexes; SortMerge merges on sorted
runs — are forthcoming but not yet implemented.

When those strategies land, this error will be downgraded to a
warning (W305) for predicates that the new strategies can execute,
leaving E313 for predicates no strategy can handle.
