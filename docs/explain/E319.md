# Error E319: Combine `on_miss: error` had no matching build row

## What it means

A combine node with `on_miss: error` evaluated a driver record that
matched no build-side row. Under `on_miss: error` the combine is
declaring that *every* driver record must have at least one match —
an unmatched driver is treated as a data-integrity violation, so the
combine aborts the run on the first miss rather than emit a partially-
populated record or skip the driver silently.

Semantically distinct from E310 (memory-budget overflow) and from the
other miss policies that emit nulls (`on_miss: null_fields`, the default)
or silently drop the driver (`on_miss: skip`).

## Example

```yaml
# Driver row whose join key isn't present on the build side
nodes:
  - type: combine
    name: enrich_orders
    input:
      orders: orders
      customers: customers
    where: "orders.customer_id == customers.id"
    match: first
    on_miss: error              # abort on the first unmatched driver
    cxl: |
      emit orders.id, customers.name
```

If an `orders.customer_id` value doesn't appear in `customers.id`, the
run aborts with:

```
E319 combine 'enrich_orders': on_miss: error — no matching build row
for driver row 142
```

`driver_row` is the 0-based ordinal of the offending record in the
driver stream — useful for locating the bad row in the source file.

## How to fix

Pick whichever matches your intent:

1. **Driver data is dirty**, and unmatched rows should be tolerated:
   change `on_miss:` to `null_fields` (build-side columns resolve to
   `null` and the driver record is still emitted) or `skip` (the
   driver record is dropped from the output).

2. **Build data is incomplete**: backfill the missing keys upstream
   (a `transform` lookup against a reference file, or a separate
   pre-load step) so every driver record has a matching build row
   before the combine runs.

3. **Both sides are authoritative**: the miss is the bug, and
   aborting is the right behavior — investigate the offending row
   (`driver_row` indicates the source position) and fix the source.

## Technical context

`on_miss: error` is the strictest of the three miss policies. The
combine executor returns `PipelineError::CombineMissingMatch` from the
probe loop the moment a driver record fails to match any build row;
the run-loop propagates that error upward and the process exits with
a non-zero status. No partial output is written for the offending
driver, and downstream operators see no record for that row.

The diagnostic carries the driver record's positional ordinal so a
user can grep their source file (`head -n 142 input.csv`) without
needing schema introspection.
